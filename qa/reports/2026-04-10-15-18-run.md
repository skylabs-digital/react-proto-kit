## Findings

### F1 — LOW — `createFallbackSeedConfig` name is misleading

**Where:** [src/helpers/seedHelpers.ts:27-36](src/helpers/seedHelpers.ts#L27-L36)

**What:** The helper is documented as "only works with 204 responses (for fetch connector)" and sets `initializeEmpty: false`. The name "fallback" suggests "use as a default when storage is empty" — which is actually what `createInitSeedConfig` does. I wired the QA sandbox to `createFallbackSeedConfig` on first read of the public API and got zero seed rows in localStorage, because the helper is silently a no-op against `LocalStorageConnector`.

**Evidence:** First sandbox run showed `localStorage.api_client_data = undefined`, lists empty, despite the provider being configured with seed data.

**Fix applied in sandbox:** switched to `createDevSeedConfig` ([qa/sandbox/src/App.tsx:19](qa/sandbox/src/App.tsx#L19)).

**Suggested library fix:** rename `createFallbackSeedConfig` → `createFetch204FallbackSeedConfig` (or add a prominent `@remarks` note in the JSDoc) so the next reader doesn't fall in the same hole. Alternatively, have it throw a dev-time warning when used with the localStorage connector.

### F2 — MEDIUM — `ErrorResponse` has three overlapping discriminators, all optional

**Where:** [src/types/index.ts:13-20](src/types/index.ts#L13-L20)

```ts
export interface ErrorResponse {
  success: false;
  message?: string;
  error?: { code: string };              // discriminator 1
  type?: 'AUTH' | 'VALIDATION' | ...;    // discriminator 2
  validation?: Record<string, string>;   // discriminator 3 (implicit)
  data?: Record<string, unknown>;
}
```

**What:** Every field after `success: false` is optional. A consumer narrowing an `ApiResponse<T>` on `!res.success` still can't rely on `res.error`, `res.type`, or `res.validation` existing. Worse, the "is this a validation error?" question has three possible sources of truth:

- `res.type === 'VALIDATION'` (top-level)
- `res.error?.code === 'VALIDATION_ERROR'` (nested, loose string)
- `res.validation && Object.keys(res.validation).length > 0`

The runtime I captured for a Zod-failed create returned **all three at once**:

```json
{
  "success": false,
  "message": "Validation failed",
  "error": { "code": "VALIDATION_ERROR" },
  "type": "VALIDATION",
  "validation": { "title": "title required" }
}
```

`error.code: string` is a loose string (no literal union), so IDE autocomplete doesn't guide users toward the valid codes listed in [docs/ERROR_HANDLING.md](docs/ERROR_HANDLING.md).

**Why it matters:** docs/ERROR_HANDLING.md is internally consistent, but the type surface forces consumers into defensive null checks and gives them three ways to write the same `if` — which means three different consumer codebases will each pick a different one and none of them will agree.

**Suggested fix (v2.1, non-breaking):**
1. Narrow `error.code` to a string literal union (`'VALIDATION_ERROR' | 'AUTH_ERROR' | 'NETWORK_ERROR' | ...`) and export it as `ApiErrorCode`.
2. Make `error` required when `success: false`, so the narrowing is reliable.
3. Keep `type` / `validation` as legacy accessors but mark them `@deprecated` in favor of `error.kind` + `error.fields`.

A v2.1 cleanup isn't breaking if the existing fields stay emitted for one minor cycle.

### F3 — MEDIUM — Delete mutation response omits `data` field but type is `ApiResponse<void>`

**Where:** [src/types/index.ts:189](src/types/index.ts#L189), runtime shape observed on `/mutations`

**What:** `DeleteMutation.mutate` is typed as `Promise<ApiResponse<void>>`, where `ApiResponse<T> = SuccessResponse<T>` has `data: T` as a required field. At runtime, delete returns:

```json
{ "success": true, "message": "Item deleted successfully" }
```

— with **no `data` key at all**. Consumers writing `'data' in res` or destructuring `{ data }` will disagree with the type definition. It's minor because `T = void` means nobody reads `res.data`, but the contract is technically violated.

**Suggested fix:** either emit `data: undefined` from the delete path to match the type, or change the type to `ApiResponse<void | undefined>` / a narrower `DeleteSuccessResponse` without `data`.

### F4 — HIGH — `withQuery()` is a silent no-op against `LocalStorageConnector`

**Where:** [src/hooks/useList.ts:157](src/hooks/useList.ts#L157) and [src/connectors/LocalStorageConnector.ts:243](src/connectors/LocalStorageConnector.ts#L243)

**What:** `useList` merges query params into the request:

```ts
const mergedParams = { ...params, ...options?.queryParams };
```

and passes them to `connector.get(endpoint, mergedParams)`. `LocalStorageConnector.get` only honours filters placed under `params.filters.*`:

```ts
if (params?.filters) {
  filteredData = filteredData.filter(item => {
    return Object.entries(params.filters).every(([key, value]) => ...);
  });
}
```

So `todosApi.withQuery({ status: 'done' }).useList()` flows `{ status: 'done' }` at the top level of `params`, which the connector **does not inspect at all**. The hook triggers a refetch cycle on every `queryParams` change, the UI re-renders, the cache key changes — and the data is identical every time. It is the textbook silent failure.

**Evidence:** on `/queries`, setting the status filter to "done" still returns all 3 seeded todos. Reproduced in a headless regression test below.

**Why it matters:** every example and tutorial I reviewed in Phase 4 recommends localStorage for local dev. Consumers who prototype filtering locally see it "work" (refetch fires, loading briefly flips) and ship code that only actually filters once the fetch connector is swapped in. Those filters will silently mis-behave in tests that use localStorage seed fixtures too.

**Suggested fix (pick one):**
- Have `useList` / the factory place query-param filters under `params.filters` so the connector picks them up uniformly, and keep top-level params for things like pagination / sort.
- Or, make `LocalStorageConnector.get` apply top-level params as equality filters after the explicit `filters` block (with a documented precedence).
- At the very least, emit a `console.warn` the first time a `withQuery(...).useList()` is issued against `LocalStorageConnector` so the silent failure becomes a loud one.

**Regression test:** [qa/generated-tests/withquery-localstorage-filtering.generated.test.tsx](qa/generated-tests/withquery-localstorage-filtering.generated.test.tsx) — currently FAILS with `expected length 1 but got 3`.

### F5 — HIGH — URL hooks don't rewrite the URL when the incoming value is invalid

**Where:** [src/navigation/useUrlTabs.ts](src/navigation/useUrlTabs.ts) and [src/navigation/useUrlStepper.ts](src/navigation/useUrlStepper.ts)

**What:** Load `http://localhost:5180/url-nav?tab=nonexistent&step=ghost`. The hooks emit a correct console warning and internal state falls back to the defaults (`tab = overview`, `step = personal`). **But `location.search` stays `?tab=nonexistent&step=ghost`.** UI now lies about URL; URL now lies about UI.

Consequences:
- Bookmarks / shared links stay broken forever — next person to open the URL sees the same warning and the same fallback, with no self-healing.
- Navigation (`nav-home → back`) brings the user back to the invalid URL.
- Analytics / query-string-based routing misreads the "real" tab.
- The hook contract is "URL is the source of truth," but after any invalid load there are two sources of truth that disagree.

**Evidence:** reproduced on `/url-nav`, confirmed in the regression test below. The warnings fire under React StrictMode once per hook per render pass (the doubling is cosmetic, but worth noting — see F7).

**Suggested fix:** on first render, if the URL value is not in the allowed set, call `setSearchParams` (replace, not push, so history isn't polluted) to normalise the URL to the default value.

**Regression test:** [qa/generated-tests/url-fallback-rewrite.generated.test.tsx](qa/generated-tests/url-fallback-rewrite.generated.test.tsx) — currently FAILS for both `useUrlTabs` and `useUrlStepper`.

### F6 — MEDIUM — Mutation scenarios are painfully easy to miswire against `useUpdate` (PUT) when you want "patch a single field"

**Where:** scenario-authoring observation, not a library bug per se, but evidence of an ergonomics gap.

**What:** In the QA sandbox I initially wired the todo-toggle button to `useUpdate(id, { title: '', status: next })`. Zod validation (`title.min(1)`) immediately rejected it with a `VALIDATION_ERROR`, and the UX read as "toggle is broken". The real answer is `usePatch`. In practice: developers following the factory's `useUpdate` signature will frequently want to change one field and be surprised that their "partial" update triggers validation errors for untouched required fields.

**Why it matters:** the factory exposes both `useUpdate` (PUT — full replace) and `usePatch` (PATCH — partial) but there is no JSDoc steering consumers between them. The error the user hits is NOT a type error (because Zod's parser runs at mutation time, not compile time), it's a runtime validation error that looks like a bug in the toggle.

**Suggested fix:** add a 1-line `@remarks` on `useUpdate` and `usePatch` at [src/factory/createDomainApi.ts:169](src/factory/createDomainApi.ts) that says "Use `useUpdate` for full replace (PUT), `usePatch` for partial updates (PATCH)."

### F7 — LOW — URL hooks emit StrictMode-doubled console warnings

**Where:** [src/navigation/useUrlTabs.ts:15](src/navigation/useUrlTabs.ts#L15), [src/navigation/useUrlStepper.ts:15](src/navigation/useUrlStepper.ts#L15)

**What:** Warnings like `[useUrlTabs] Invalid value "nonexistent" ...` fire twice per render under React StrictMode (once per pass). Cosmetic noise that makes finding the first occurrence harder in crowded logs.

**Suggested fix:** guard the `console.warn` with a ref that remembers "I already warned about this (key, value) pair this render cycle". Or leave it as-is and document in a FAQ.
